Madcap Flare setup and overview

Madcap Flare is a content management system (CMS) that is used to write and publish content in a wide variety of formats without altering the source. This is called single-sourcing.

Flare is completely XML-based, from the content structuring itself, right down to the layout and publishing components. This means that all the files that you'll create and use for publishing are static—that is, you do not connect to a database or shared file system and all the files in your project reside on your local drive. This presents certain obstacles to maintaining consistency across your projects because: (a) changes made to the layout in one project are not updated in the others; and (b) changes made in the same project by one writer are not updated in another writer's local copy. The latter is easily resolved using a source-control system like Github or SVN. Read more about Source Control in Flare.

Source control takes care of intra-project synchronization, but what about inter-project maintenance of the stylistic and publishing components? Flare officially supports a global synchronization option with a master (or parent) project. See Generating a Consolidated Output from Multiple MadCap Flare Projects and Single-sourcing Best Practices with Multiple MadCap Flare Projects for more information.

General overview

A full overview or introduction is beyond the scope of this style guide. Read the Getting Started with MadCap Flare online guide.

The key points for those familiar with other CMSs are as follows:

  • You write and save your content to HTML files. The schema follows some version of the HTML5 standard with proprietary XML tags for variables, conditions, and snippets.
  • The HTML (and other) tags are styles by a subset of the CSS3 standard, so not all elements and properties are supported. Additionally, Flare-specific elements are available, mostly for PDF styling.
  • You design the layout of print outputs like PDF and WORD visually using Page Layouts in the layout editor. You can use CSS styles to complement the design of the layout editor.

Good housekeeping

Follow these best practices for organizing your files and content:

  • Images

    • Use SVGs for your vector illustrations as this format is natively supported in Flare. See Appendix A SVG compatibility and export settings for more information.

      Always save your source file in PDF format and export to SVG format.

    • Use camel-case for naming your images in a descriptive way and prepend i_ or pic_ to your vector and raster files respectively. For example, i_GC_engineTuningMeasureRPM.svg and pic_WA_XMDAppAddPoint.jpg.

    • Style your images using the predefined styles in the stylesheet.

  • Topic files
    • In projects with multiple products, prepend the product's code to the topic's file name. For example, WA_Safety_Alarms.htm.
      For shared topics, prepend zgen_ to the topic's file name. For example, zgen_General_Safety.htm.
    • Prepend the topic type for chapter (CH_) and appendix (APPX_) topics. For example, WA_APPX_Wear_Limits.htm.
  • Snippets

    • Save your snippets in the Topics folder and not in the Snippets folder.

    • Prepend zsnp_ to snippet files. For example zsnp_WA_procTuning.flsnp.

      Add a descriptive abbreviation of the type of topic (like "proc" for procedure).

  • Alerts
    • Save your alerts as snippets when they can be reused.
    • Follow a similar naming convention as described above for your snippet files.
